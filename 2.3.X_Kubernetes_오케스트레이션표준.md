#### 2.3.X Kubernetes — 배포 환경 공통 오케스트레이션 표준

배포 환경(클라우드·온프레미스·하이브리드)이 달라져도, AI Agent 시스템을 구성하는 대부분의 기술(모델 서빙, Vector DB, Observability 등)은 **컨테이너 기반 배포와 Kubernetes 오케스트레이션을 전제**로 설계되어 있다. CNCF 2025 조사에서도 Kubernetes가 AI/ML 워크로드의 사실상(de facto) 실행 기반으로 자리잡았음이 확인되었다.

따라서 배포 환경 유형을 결정한 뒤, **해당 환경에서 Kubernetes를 어떤 형태로 제공받을 것인지**를 다음 의사결정 항목으로 확정할 것을 권장한다. 이후 3장~10장의 기술 선정은 Kubernetes 위에서의 운영을 전제로 한다.

##### 환경별 Kubernetes 제공 서비스 매핑

| 배포 환경 | K8s 서비스 | 특성 | GPU 지원 |
| :---- | :---- | :---- | :---- |
| **AWS** | EKS (Elastic Kubernetes Service) | 관리형 컨트롤 플레인, Karpenter 기반 GPU 노드 오토스케일링 | p5/p4d(H100/A100) 인스턴스, EKS GPU AMI |
| **Azure** | AKS (Azure Kubernetes Service) | Azure OpenAI Service와 네이티브 통합, GPU 노드풀 | NC/ND 시리즈(A100/H100), NVIDIA GPU Operator |
| **GCP** | GKE (Google Kubernetes Engine) | Autopilot 모드 지원, TPU/GPU 스케줄링 내장 | A3/G2(H100/L4) 머신 타입, GKE GPU 자동 프로비저닝 |
| **온프레미스** | kubeadm / RKE2 / k3s 등 자체 구축 | 컨트롤 플레인·워커 노드 직접 운영, 업그레이드 책임 증가 | NVIDIA GPU Operator + Device Plugin 필수 설치 |
| **하이브리드** | 위 조합 + 멀티클러스터 관리 | Karmada·Crossplane 등으로 클러스터 간 워크로드·정책 통합 | 환경별 GPU 스케줄링 각각 구성, 통합 모니터링 필요 |

> **예외:** Docker Compose 등으로 충분한 소규모 PoC·로컬 개발 환경에서는 Kubernetes 도입이 과도할 수 있다. 프로덕션 전환 시점에 K8s 기반으로 이행하는 것을 권장한다.
