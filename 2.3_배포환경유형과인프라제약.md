## 2.3 배포 환경 유형과 인프라 제약

AI Agent의 기술 선정은 **"어떤 기술을 고르는가"**뿐 아니라 **"어디에서 실행하는가"**에 의해서도 크게 달라진다.  
배포 환경은 3장~10장 전체에 걸쳐 선택지를 제한하는 **횡단 조건(Cross-cutting Constraint)**이므로, 개별 영역 선정에 앞서 환경 유형을 먼저 확정해야 한다.

환경 유형이 달라지면 동일한 기능 요구사항이라도 **선택 가능한 기술 후보군이 근본적으로 달라질 수 있기 때문**이다.

---

# 배포 환경 유형

AI Agent 시스템의 배포 환경은 아래 **네 가지 유형**으로 구분한다.

| 유형 | 정의 | 핵심 키워드 |
|---|---|---|
| **Public Cloud** | AWS·Azure·GCP 등 외부 사업자의 관리형 인프라 사용 | 종량제, 관리형 서비스, 상용 LLM API |
| **Private Cloud** | 자체 데이터센터에 OpenShift 등 컨테이너 플랫폼을 구축하여 Cloud-like 운영 | 데이터 격리, K8s 오케스트레이션, 셀프호스팅 |
| **On-Premise** | 베어메탈·VM 기반 자체 인프라를 직접 구축·운영 (컨테이너 플랫폼 없음) | 폐쇄망, 수동 운영, 최소 외부 의존 |
| **Hybrid** | 데이터 민감도에 따라 Public Cloud + Private Cloud / On-Premise를 혼합 | 워크로드 분리, 데이터 등급 기반 라우팅 |

---

# 배포 환경 유형 판단 기준

프로젝트 착수 시 아래 **5단계 질문**을 순서대로 확인하여 환경 유형의 방향을 결정한다.  
각 단계는 이전 단계의 결과를 입력으로 받으므로, **1단계부터 순차적으로 진행**한다.

### 1단계 – 데이터 반출 가능 여부 (최우선 판단)

AI Agent는 사용자 질의·문서·업무 데이터를 LLM에 전달하므로, **처리 대상 데이터의 외부 전송 가능 여부**가 환경 유형을 가장 크게 좌우한다.

| 질문 | 답변 | 판단 방향 | 기술적 근거 |
|---|---|---|---|
| AI Agent가 처리하는 데이터를 외부 클라우드로 전송할 수 있는가? | **전면 가능** | → 2단계로 이동 | 상용 LLM API(GPT, Claude, Gemini)를 직접 호출할 수 있어 **모델 서빙 인프라 구축 부담이 없음** |
| | **전면 불가** | → Private Cloud 또는 On-Premise 방향, 3단계로 이동 | 개인정보보호법·금융위 클라우드 이용 가이드·ISMS-P·국방 보안규정 등에 의해 외부 전송이 금지되는 경우, **오픈소스 LLM 셀프호스팅이 필수**이며 GPU 인프라를 내부에 확보해야 함 |
| | **일부 가능** | → Hybrid 방향, 4단계로 이동 | 데이터 등급(공개/내부/기밀)에 따라 비민감 질의는 상용 API로, 민감 질의는 내부 모델로 분기하는 **라우팅 레이어 설계**가 필요 |

### 2단계 – 인프라 소유 및 운영 전략 (데이터 반출 가능 시)

외부 전송이 가능하더라도, 이미 보유한 GPU 자산이나 장기 비용 효율에 따라 자체 인프라 활용이 유리할 수 있다.

| 질문 | 답변 | 판단 방향 | 기술적 근거 |
|---|---|---|---|
| 자체 데이터센터·GPU 인프라를 보유하고 있거나 신규 투자 계획이 있는가? | **없음** | → **Public Cloud** 우선 검토 | Bedrock·Azure OpenAI·Vertex AI 등 관리형 서비스를 사용하면 **GPU 프로비저닝·모델 배포·오토스케일링을 사업자가 처리**하므로 Time-to-Market이 가장 짧음 |
| | **있음** | → Public Cloud 또는 Hybrid 비교 검토 | A100/H100 등 고가 GPU를 이미 보유한 경우, LLM 추론 워크로드를 내부에서 처리하면 **토큰당 비용이 상용 API 대비 절감**될 수 있음. 다만 모델 업데이트·서빙 파이프라인·장애 대응 등 **운영 오버헤드**를 함께 비교해야 함 |

### 3단계 – 컨테이너 플랫폼 유무 (데이터 반출 불가 시)

데이터 반출이 불가한 환경에서 AI Agent를 운영하려면, 모델 서빙·Vector DB·Observability 등 **여러 컴포넌트를 내부에 직접 배포**해야 한다.  
이때 컨테이너 오케스트레이션 플랫폼의 유무가 운영 복잡도와 확장성을 결정짓는 핵심 분기점이 된다.

| 질문 | 답변 | 판단 방향 | 기술적 근거 |
|---|---|---|---|
| OpenShift·Kubernetes 등 컨테이너 플랫폼을 운영 중이거나 도입 가능한가? | **예** | → **Private Cloud** 우선 검토 | K8s 기반 GPU 스케줄링(NVIDIA GPU Operator), Operator 패턴에 의한 **모델 서빙·DB·모니터링의 선언적 배포**, HPA/VPA를 활용한 **Pod 단위 오토스케일링** 등 Cloud-like 운영이 가능 |
| | **아니오** | → **On-Premise** 방향 검토 | 베어메탈·VM 위에 vLLM/TGI를 직접 설치·운영하는 구조. **배포 자동화·롤링 업데이트·장애 복구가 수동**에 가까우므로, 소규모·고정 사용자 환경에 적합 |

### 4단계 – SLA·확장성 요구 수준

환경 유형 방향이 정해진 뒤, 실제 서비스 수준 목표에 따라 구체적인 인프라 구성을 조정한다.

| 질문 | 답변 | 권장 환경 | 기술적 근거 |
|---|---|---|---|
| 서비스 가용성(SLA) 및 트래픽 탄력성 요구 수준은? | **SLA 99.9%↑, 트래픽 급변 대응 필요** | Public Cloud 관리형 서비스 권장 | Multi-AZ 자동 Failover, 관리형 LLM 엔드포인트의 **수 초 내 오토스케일링**, 사업자 SLA 보장 활용 가능 |
| | **SLA 99.5% 수준, 트래픽 예측 가능** | Private Cloud 적합 | OpenShift HPA 기반 Pod 스케일링으로 대응 가능하나, **스케일 상한은 물리 노드 수에 종속**되므로 사전 용량 계획 필요 |
| | **내부 SLA, 소규모 고정 사용자** | On-Premise 허용 | 단일 GPU 서버에서 vLLM/Ollama로 서빙하는 **단순 구성**으로도 충분. 다만 **단일 장애점(SPOF)**에 대한 수용이 전제됨 |

### 5단계 – 조직 운영 역량 및 비용 구조 최종 검증

앞 단계에서 도출된 방향이 조직의 현실적 역량과 예산 구조에 부합하는지 최종 점검한다.  
부합하지 않으면 **이전 단계로 돌아가 대안을 재검토**한다.

| 질문 | 답변 | 고려사항 |
|---|---|---|
| K8s/OpenShift 운영 경험과 GPU 관리 인력이 확보되어 있는가? | **미확보** | Private Cloud·On-Premise를 선택할 경우 **GPU 드라이버·CUDA 버전 관리, 모델 양자화(Quantization) 튜닝, 서빙 파이프라인 장애 대응** 등의 운영 부담이 크므로, Public Cloud로 재검토하거나 외부 운영 위탁(Managed Service)을 함께 고려 |
| | **확보** | Private Cloud·On-Premise 운영 가능. 다만 **모델 업데이트 주기, GPU 노드 증설 계획, 클러스터 업그레이드 전략** 등 중장기 운영 로드맵을 사전에 수립해야 함 |
| 예산 구조가 CapEx(선투자) 중심인가, OpEx(종량제) 중심인가? | **OpEx 선호** | Public Cloud가 유리. GPU 인스턴스 비용은 높지만 **예약 인스턴스·Savings Plan으로 40~60% 절감** 가능하며, 사용량 변동에 따라 유연하게 조정 |
| | **CapEx 가능** | Private Cloud·On-Premise가 가능. 초기 GPU 서버 투자 비용이 높으나 **3년 이상 장기 운영 시 토큰당 단가가 상용 API 대비 유리**해질 수 있음 (TCO 분석 권장) |

### 판단 흐름 요약

```
1. 데이터 반출 가능?
   ├─ 전면 가능 → 2. 자체 GPU 인프라?
   │                 ├─ 없음 ──────→ [Public Cloud 우선 검토]
   │                 └─ 있음 ──────→ [Public Cloud] 또는 [Hybrid] 비교 검토
   │
   ├─ 전면 불가 → 3. K8s/OpenShift 도입 가능?
   │                 ├─ 예 ────────→ [Private Cloud 우선 검토]
   │                 └─ 아니오 ────→ [On-Premise 방향 검토]
   │
   └─ 일부 가능 → [Hybrid 방향 검토]
                    (민감 → Private/On-Premise, 비민감 → Public Cloud)

   ※ 모든 경로는 4단계(SLA·확장성) → 5단계(운영 역량·비용) 검증을 거쳐 최종 결정
```

# Public Cloud 환경 (AWS, Azure, GCP)

**특성**

- 관리형 서비스 활용 가능, 자동 스케일링 지원
- 초기 투자 없이 종량제(OpEx) 사용

**기술 선정 영향**

- 상용 LLM API 활용 가능
  - AWS → **Amazon Bedrock**
  - Azure → **Azure OpenAI**
  - GCP → **Vertex AI**
- Cloud Native 서비스와의 연동 용이

**고려사항**

- 데이터가 외부 클라우드로 전송되므로  
  **데이터 주권(Data Sovereignty) 및 규제 요건 확인 필요**

**적합 시나리오**

- PoC, 글로벌 서비스, 탄력적 트래픽 대응 서비스

> 사업자별 세부 비교는 `2.3_클라우드환경_사업자별세분화.md` 참조

# Private Cloud 환경 (OpenShift 기반)

**특성**

- 자체 데이터센터에 Kubernetes(OpenShift) 플랫폼을 구축하여 **Cloud-like 운영**
- 데이터가 자체 인프라에 유지되어 **외부 유출 원천 차단**
- Operator 패턴·GitOps 등 선언적 관리 가능

**기술 선정 영향**

- LLM → 오픈소스 모델 셀프호스팅 필수
  - vLLM / TGI를 **OpenShift AI(RHOAI)** 또는 Pod로 배포
- Vector DB → Milvus·Qdrant·Weaviate 등을 Operator / Helm으로 배포
- Observability → OpenShift 내장 모니터링(Prometheus/Grafana) + Langfuse 셀프호스팅

**고려사항**

- **NVIDIA GPU Operator**로 GPU 노드를 클러스터에 추가·관리 (하드웨어 조달 리드타임 발생)
- OpenShift 서브스크립션 비용 + K8s 운영 인력 확보 필요
- SCC(Security Context Constraints) 등 **기본 보안 정책이 엄격** → 이미지·권한 설정 시 추가 작업
- 스케일링 상한이 물리 자원에 종속 → **용량 계획(Capacity Planning)** 필수

**적합 시나리오**

- 금융·공공·국방·의료 등 데이터 외부 반출 불가 환경
- OpenShift 클러스터 또는 Red Hat 기술 지원 계약을 이미 보유한 조직
- 폐쇄망이지만 Cloud-like 운영(GitOps, 자동 스케일링)을 원하는 경우

# On-Premise 환경

**특성**

- 데이터 외부 반출 없음
- 폐쇄망 운영 가능
- 인프라 직접 구축 및 운영 필요

**기술 선정 영향**

- LLM → **오픈소스 모델 셀프호스팅 필수**
  - LLaMA
  - Mistral
  - Mixtral
- Vector DB → **셀프호스팅 가능 제품으로 제한**
  - Milvus
  - Qdrant
  - Weaviate
- Observability → **셀프호스팅 도구 사용**
  - Langfuse
  - Arize Phoenix

**고려사항**

- GPU 서버 확보 필요
- 모델 서빙 인프라 구축 필요
  - vLLM
  - TGI
  - Ollama
- 운영 인력 확보 필요

**적합 시나리오**

- 금융
- 의료
- 공공기관
- 국방
- 제조 폐쇄망 환경

# 하이브리드 환경

**특성**

- 민감 데이터 → 온프레미스
- 일반 데이터 → 클라우드

즉 **데이터 민감도 기반 워크로드 분리 구조**

**기술 선정 영향**

- 데이터 등급 기반 LLM 호출 분기


**고려사항**

- 데이터 분류 기준 사전 정의
- 보안 연결 구성 필요

예시

- VPN
- 전용선 (Direct Connect / ExpressRoute)
- PrivateLink

또한

- 네트워크 지연 시간
- 데이터 전송 정책
- 보안 규정

을 고려한 설계 필요

**적합 시나리오**

- 데이터 민감도 기반 워크로드 분리가 필요한 기업
- 금융/제조/대기업 내부 시스템

# 배포 환경별 기술 스택 비교

| 기술 영역 | Public Cloud | Private Cloud (OpenShift) | On-Premise | Hybrid |
|---|---|---|---|---|
| **LLM (3장)** | 상용 API (GPT, Claude, Gemini) | 오픈소스 셀프호스팅 (LLaMA, Mistral) – RHOAI / vLLM Pod | 오픈소스 셀프호스팅 + GPU 서버 직접 관리 | 민감 → Private/OnPrem 모델 / 비민감 → 상용 API |
| **Data Store (7장)** | 관리형 (Aurora pgvector, OpenSearch, AI Search, Vertex AI Vector Search) | Operator/Helm 배포 (Milvus, Qdrant, Weaviate) | 셀프호스팅 전용 (Milvus, Qdrant, Weaviate) | 민감 → 로컬 / 일반 → 클라우드 분산 저장 |
| **Observability (10장)** | SaaS (LangSmith, Helicone) | OpenShift 내장 모니터링 + Langfuse 셀프호스팅 | 셀프호스팅 (Langfuse, Arize Phoenix) | 프록시 기반 비식별화 후 전송 |
| **Embedding (7장)** | 상용 API (OpenAI, Cohere) | 오픈소스 셀프호스팅 (BGE-M3, HuggingFace) | 오픈소스 + 셀프호스팅 | 성능·보안 요구에 따라 선택 |

# 오픈소스 모델 셀프호스팅 최소 인프라 요구사항 (Cloud 기준)

| 모델 크기 | 파라미터 규모 | 필요 VRAM (4-bit 기준) | AWS 추천 인스턴스 | Azure 추천 인스턴스 | 비고 |
|---|---|---|---|---|---|
| 소형 모델 | 7B ~ 8B | 8GB ~ 12GB | g5.xlarge (A10G 24GB ×1) | Standard_NC4ads_A10_v4 (A10 24GB ×1) | 개인 비서, 간단한 요약 |
