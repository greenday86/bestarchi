### 2.3 배포 환경 유형과 인프라 제약

AI Agent의 기술 선정은 "어떤 기술을 고르는가"뿐 아니라 **"어디에서 실행하는가"**에 의해서도 크게 달라진다. 배포 환경은 3장~10장 전체에 걸쳐 선택지를 제한하는 **횡단 조건(Cross-cutting Constraint)**이므로, 개별 영역 선정에 앞서 환경 유형을 먼저 확정해야 한다. 환경 유형이 달라지면 동일한 기능 요구사항이라도 선택 가능한 기술 후보군이 근본적으로 바뀌기 때문이다.

#### 2.3.1 배포 환경 유형 판단 기준

프로젝트 착수 시 아래 세 가지 질문에 먼저 답해야 한다. 이 답에 따라 환경 유형이 결정되며, 이후 3장~10장의 기술 선정 범위가 한정된다.

| 판단 질문 | 답변 예시 | 결정되는 환경 유형 |
| :---- | :---- | :---- |
| 처리 대상 데이터가 외부 클라우드로 전송 가능한가? | 모두 가능 → 클라우드 / 전면 불가 → 온프레미스 / 일부만 가능 → 하이브리드 | 1차 분류 |
| 자체 GPU 인프라를 확보·운영할 역량이 있는가? | 없음 → 클라우드 우선 / 있음 → 온프레미스·하이브리드 가능 | 2차 확인 |
| 서비스 가용성·확장성 요구 수준은? | SLA 99.9% 이상·자동 스케일링 필수 → 클라우드 관리형 / 내부 SLA로 충분 → 온프레미스 허용 | 3차 확인 |

#### 2.3.2 클라우드 환경 (AWS, GCP, Azure 등)

| 항목 | 내용 |
| :---- | :---- |
| **특성** | 관리형 서비스 활용 가능, 자동 스케일링, 인프라 운영 부담 최소화. 초기 투자 없이 종량제(pay-as-you-go)로 시작 가능 |
| **기술 선정 영향** | 상용 LLM API 즉시 사용 가능(GPT-5.x, Claude 4.x, Gemini 3.x). 클라우드 네이티브 AI 플랫폼(Amazon Bedrock, Azure OpenAI Service, Vertex AI) 활용으로 모델 호스팅·스케일링·모니터링 통합 가능. 관리형 Vector DB(Pinecone, Amazon OpenSearch Serverless 등), 관리형 Observability(LangSmith, Helicone) 연동 용이 |
| **주요 고려사항** | 데이터가 외부 클라우드로 전송되므로 **데이터 주권·규제 요건**(개인정보보호법, 금융권 클라우드 이용 가이드라인 등) 확인 필수. 리전 선택에 따른 지연 시간 차이. 비용 증가 시 Model Routing(3장)으로 최적화 |
| **적합 시나리오** | 데이터 규제 제약이 낮은 프로젝트, 빠른 프로토타이핑 및 MVP, 글로벌 서비스, 탄력적 트래픽 대응이 필요한 프로덕션 |

#### 2.3.3 온프레미스 환경

| 항목 | 내용 |
| :---- | :---- |
| **특성** | 데이터가 외부로 나가지 않음(폐쇄망 운영 가능). 인프라 직접 구축·운영 필요. 초기 투자(GPU 서버, 네트워크 등)가 크나 장기 운영 시 단위 비용 절감 가능 |
| **기술 선정 영향** | LLM은 오픈소스 셀프호스팅 **필수**(LLaMA 3.x, Mistral, DeepSeek V3.x 등). 모델 서빙 인프라(vLLM, TGI, Ollama 등) 별도 구축. Vector DB는 셀프호스팅 가능 제품으로 제한(Milvus, Qdrant, Weaviate, pgvector). Embedding 모델도 오픈소스(BGE-M3 등) 셀프호스팅. Observability는 셀프호스팅 도구(Langfuse, Arize Phoenix, OpenTelemetry+SigNoz) 선택 |
| **주요 고려사항** | GPU 서버 확보(NVIDIA H100/A100 또는 L40S 이상 권장), 모델 서빙·오케스트레이션 운영 인력 필요. 모델 업데이트·패치를 자체적으로 관리해야 하므로 MLOps 역량 요구 |
| **적합 시나리오** | 금융·의료·공공 등 데이터 외부 전송이 불가한 규제 산업, 대규모 처리량으로 클라우드 API 비용 분기점을 초과하는 경우, 모델 커스터마이징(LoRA, Adapter)이 필수인 프로젝트 |

#### 2.3.4 하이브리드 환경

| 항목 | 내용 |
| :---- | :---- |
| **특성** | 민감 데이터는 온프레미스에서, 비민감 처리는 클라우드에서 수행. 두 환경의 장점을 결합하되 설계 복잡도가 증가 |
| **기술 선정 영향** | 데이터 분류 등급에 따라 LLM 호출 경로가 분기(민감 → 온프레미스 오픈소스 모델, 비민감 → 클라우드 상용 API). API Gateway 또는 Model Router에서 분류 등급 기반 라우팅 구현 필요. 온프레미스↔클라우드 간 네트워크 지연 시간 설계, 장애 시 Fallback 정책 필요 |
| **주요 고려사항** | **데이터 분류 기준 사전 정의**(민감/비민감 경계, 분류 자동화 방안). 온프레미스↔클라우드 간 **통신 보안**(VPN, Private Link, 전용선). 두 환경의 Observability 통합(분산 트레이싱 연결). 환경별 장애 격리 및 Fallback 전략 |
| **적합 시나리오** | 규제 데이터와 일반 데이터가 혼재하는 프로젝트, 온프레미스 모델로 비용을 절감하면서 복잡 추론만 클라우드 플래그십 모델을 활용하려는 경우 |

#### 2.3.5 배포 환경이 기술 선정에 미치는 영향 요약

3장~10장 각 기술 영역에서 배포 환경별로 선택지가 어떻게 달라지는지를 한눈에 정리한다. 프로젝트 착수 시 이 표를 먼저 확인하여 선택 범위를 한정한 후 각 장의 상세 비교로 진입하는 것을 권장한다.

| 기술 영역 | 클라우드 | 온프레미스 | 하이브리드 |
| :---- | :---- | :---- | :---- |
| **LLM (3장)** | 상용 API (GPT-5.x, Claude 4.x, Gemini 3.x). 클라우드 네이티브 플랫폼(Bedrock, Azure OpenAI, Vertex AI) 활용 | 오픈소스 셀프호스팅 (LLaMA 3.x, Mistral, DeepSeek V3.x) + GPU 서버 + 모델 서빙(vLLM, TGI) | 민감 → 온프레미스 모델, 비민감 → 클라우드 API. Model Router에서 분류 등급 기반 분기 |
| **Framework (4장)** | 모든 Framework 사용 가능. LangGraph Platform(관리형) 활용 가능 | 동일 Framework 사용 가능하나 인프라 직접 운영 | 동일 Framework, 호출 대상 LLM만 환경별 분기 |
| **Tool 연동 (5장)** | MCP Server를 클라우드 서비스로 배포, 외부 SaaS 도구 연동 자유 | MCP Server를 내부 네트워크에 배포, 외부 SaaS 연동 불가(프록시 또는 미러링 필요) | MCP Server를 양쪽에 배포, 도구별 네트워크 접근 경로 설계 필요 |
| **Memory (6장)** | 관리형 Redis(ElastiCache), 관리형 PostgreSQL(RDS/Aurora) 활용 | 셀프호스팅 Redis, PostgreSQL. 운영·백업 직접 관리 | 환경별 저장소 분리 또는 온프레미스 중앙 집중 |
| **Data Store (7장)** | 관리형 (Pinecone, OpenSearch Serverless) 또는 셀프호스팅 | 셀프호스팅 전용 (Milvus, Qdrant, Weaviate, pgvector) | 민감 데이터 → 온프레미스 Vector DB, 비민감 → 클라우드 관리형. 용도별 분리 |
| **RAG Pipeline (8장)** | 클라우드 스토리지(S3 등)에서 직접 수집, 관리형 서비스 연동 용이 | 내부 파일서버·DB에서 수집, 외부 웹 크롤링 제한 가능 | 데이터 소스 위치에 따라 수집 경로 분리 설계 |
| **Guardrails (9장)** | 모든 도구 사용 가능 (NeMo Guardrails, Guardrails AI, LlamaGuard 등) | 동일 도구 사용 가능 (오픈소스 기반으로 셀프호스팅) | 환경별 동일 규칙 적용, 정책 동기화 필요 |
| **Observability (10장)** | 클라우드 SaaS (LangSmith, Helicone) 또는 관리형 모니터링 | 셀프호스팅 전용 (Langfuse, Arize Phoenix, OpenTelemetry+SigNoz) | 이중 구성 후 통합 대시보드로 연결. 분산 트레이싱 연결 설계 필요 |
| **Embedding (7장)** | 상용 API (OpenAI text-embedding-3, Cohere embed-v3, Voyage-3) | 오픈소스 (BGE-M3) + 셀프호스팅. GPU 또는 고성능 CPU 필요 | 혼용 가능. 단, Embedding 모델 변경 시 Vector DB 인덱스 재구축 비용 고려 |

#### 2.3.6 최소 인프라 요구사항 기준선

아래 표는 AI Agent 시스템을 운영하기 위한 최소 인프라 사양이다. 프로젝트 초기 리소스 산정 시 참고하되, 실제 요구사항은 모델 크기·동시 사용자 수·데이터 규모에 따라 달라진다.

| 구분 | 클라우드 API만 사용 | 오픈소스 모델 셀프호스팅 (7B~13B 급) | 오픈소스 모델 셀프호스팅 (70B+ 급) |
| :---- | :---- | :---- | :---- |
| CPU | 4 vCPU 이상 | 16 vCPU 이상 | 32 vCPU 이상 |
| 메모리 | 16GB RAM | 64GB RAM 이상 | 128GB RAM 이상 |
| GPU | 불필요 | NVIDIA L40S / RTX 4090 이상 (VRAM 24GB+) | NVIDIA A100 80GB 또는 H100 (멀티 GPU 구성) |
| 스토리지 | 50GB (코드, 벡터 인덱스) | 200GB+ (모델 파일, 벡터 인덱스, 로그) | 500GB+ (대형 모델 파일, 양자화 버전 포함) |
| 네트워크 | LLM API 호출을 위한 외부 통신 필수 | 외부 통신 불필요 (폐쇄망 가능) | 동일. 멀티 GPU 간 NVLink/InfiniBand 권장 |
| 모델 서빙 | 해당 없음 | vLLM 또는 TGI 단일 인스턴스 | vLLM + Tensor Parallelism 또는 TGI 분산 배포 |

#### 2.3.7 배포 환경 선택 의사결정 트리

```
[데이터 외부 전송 가능?]
    ├─ 전면 가능 ──────────── [GPU 인프라 보유?]
    │                           ├─ 없음 ────── ✅ 클라우드 (관리형 서비스 활용)
    │                           └─ 있음 ────── [비용 분기점 초과?]
    │                                            ├─ 예 ──── ✅ 클라우드 + 일부 셀프호스팅 (하이브리드)
    │                                            └─ 아니오 ── ✅ 클라우드 (API가 더 경제적)
    ├─ 일부만 가능 ─────────── ✅ 하이브리드 (데이터 분류 기반 라우팅)
    └─ 전면 불가 ──────────── ✅ 온프레미스 (폐쇄망, 오픈소스 전용)
```
