## 2.2 부록 — 클라우드 환경 사업자별 배포 유형과 인프라 제약

> 이 문서는 `2.2_배포환경유형과인프라제약.md` 2.2.7절에서 참조하는 **상세 문서**입니다.  
> 2.2절에서 Public Cloud 또는 Private Cloud로 방향을 정한 뒤, 구체적인 사업자·플랫폼을 선택할 때 활용합니다.

`2.2_배포환경유형과인프라제약.md`에서 **클라우드 환경을 하나의 유형으로 통합**해 다루었기 때문에, 여기서는 **Public Cloud(AWS, Azure, GCP)**와 **Private Cloud(OpenShift 기반)**로 나누어 배포/인프라 제약을 세분화한다.

이 문서는 다음과 같은 상황에서 활용한다.

- 클라우드 환경으로 방향은 정했으나 **Public vs Private**, 또는 **어떤 사업자를 우선 고려해야 할지** 판단할 때
- 베이스 구조는 동일하지만 **사업자별·플랫폼별 관리형 서비스/제약 차이**를 명확히 비교하고 싶을 때

---

## 공통 전제 – 클라우드 환경

- 관리형 서비스 또는 컨테이너 오케스트레이션 기반 자동 스케일링 지원
- 상용 LLM API 및 Cloud Native 서비스와의 연동 용이 (Public Cloud)
- 오픈소스 셀프호스팅 + 컨테이너 플랫폼으로 동일 기능 구현 가능 (Private Cloud)
- **데이터 위치와 규제**(데이터 주권, 국외 이전 등)에 따라 Public / Private 선택이 갈림

이 공통 전제를 유지하되, 실제 의사결정에 필요한 핵심만 **비교 표**로 압축한다.

---

## AWS / Azure / GCP 비교 표 (핵심 요약)

| 구분 | AWS | Azure | GCP |
|---|---|---|---|
| **특성** | 글로벌 확장·서비스 폭, 네트워크 옵션 다양 | MS 생태계/SSO·보안·엔터프라이즈 통합 | 데이터/ML 플랫폼 강점, K8s 친화 |
| **기술 선정 영향** | Bedrock, OpenSearch/Aurora(pgvector), CloudWatch 중심 | Azure OpenAI, Azure AI Search, Monitor/APIM 연계 | Vertex AI(생성형/에이전트), Vector Search, BigQuery 연계 |
| **인프라/운영 제약 및 고려사항** | GPU 비용 관리(스케일/예약), 리전/데이터 규제, Direct Connect/PrivateLink 설계 | Entra ID·RBAC·테넌트/권한 설계, Private Endpoint/ExpressRoute, 라이선스·플랜 영향 | 목표 리전에서 Vertex AI 등 관리형 AI 기능과 원하는 GPU 타입(A100/H100/L4 등)이 **지원·할당(쿼터) 가능한지** 사전 확인, 조직/프로젝트 구조에 따른 권한·네트워크 복잡도, 데이터 중심 설계 시 락인 고려 |
| **적합 시나리오** | 글로벌 서비스, 이벤트/분산 아키텍처, 기존 AWS 자산 보유 | M365/Windows/AD 기반 기업, 업무 자동화·엔터프라이즈 SSO 필수 | 분석·파이프라인 중심, MLOps/데이터·AI 플랫폼 일원화 |

## Private Cloud – OpenShift 기반 환경

Public Cloud가 아닌 **자체 데이터센터 또는 호스팅 IDC에 클라우드 플랫폼을 구축**하는 경우를 다룬다.  
여기서는 가장 대표적인 엔터프라이즈 Kubernetes 플랫폼인 **Red Hat OpenShift**를 전제로 정리한다.

| 구분 | OpenShift 기반 Private Cloud |
|---|---|
| **특성** | 데이터가 자체 인프라에 유지되어 **외부 유출 원천 차단**, Kubernetes 기반이지만 엔터프라이즈급 보안·인증·CI/CD가 플랫폼에 내장, Operator 패턴으로 복잡한 미들웨어의 선언적 관리 가능 |
| **기술 선정 영향** | **LLM**: 오픈소스 모델 셀프호스팅 필수 – vLLM / TGI 등을 OpenShift AI(RHOAI) 또는 직접 Pod로 배포, **Vector DB**: Milvus·Qdrant·Weaviate 등을 Operator 또는 Helm으로 배포, **Observability**: OpenShift 내장 모니터링(Prometheus/Grafana) + Langfuse 셀프호스팅 |
| **인프라/운영 제약 및 고려사항** | GPU 노드(NVIDIA GPU Operator)를 클러스터에 추가·관리해야 하며 **하드웨어 조달 리드타임** 발생, OpenShift 서브스크립션 비용 + 운영 인력 확보 필요, SCC(Security Context Constraints) 등 **기본 보안 정책이 엄격**하여 컨테이너 이미지·권한 설정 시 추가 작업 필요, 스케일링 상한이 물리 자원에 종속되므로 **용량 계획(Capacity Planning)**이 핵심 |
| **적합 시나리오** | 금융·공공·국방·의료 등 **데이터 외부 반출이 불가**한 규제 환경, 이미 OpenShift 클러스터를 운영 중이거나 Red Hat 기술 지원 계약이 있는 조직, 온프레미스이면서도 **Cloud-like 운영(GitOps, Operator, 자동 스케일링)**을 원하는 경우 |

> **Public Cloud 위의 OpenShift (ROSA / ARO / OSD)**  
> AWS(ROSA), Azure(ARO), GCP(OSD) 등 Public Cloud 위에서 관리형 OpenShift를 사용할 수도 있다.  
> 이 경우 Public Cloud의 관리형 서비스(Bedrock, Azure OpenAI 등)와 OpenShift 워크로드를 **혼합 배치**할 수 있으며,  
> 네트워크·보안 설계는 해당 사업자의 Private Endpoint/VPC 구조를 따른다.

---

## Public vs Private Cloud 선택 기준 (요약)

| 판단 기준 | Public Cloud (AWS / Azure / GCP) | Private Cloud (OpenShift) |
|---|---|---|
| 데이터 외부 전송 | 가능 (규제 확인 필요) | 불가 또는 불허 환경 |
| LLM 활용 방식 | 상용 API 우선 | 오픈소스 셀프호스팅 필수 |
| 인프라 초기 투자 | 종량제, 낮음 | 하드웨어·라이선스 선투자 필요 |
| 스케일링 | 사실상 무제한 (비용 비례) | 물리 자원 상한에 종속 |
| 운영 부담 | 관리형 서비스로 경감 | 플랫폼·GPU·모델 서빙 전반 자체 운영 |
| 보안·규제 충족 | 리전/서비스 단위로 대응 | 물리적 격리로 원천 충족 |

---

## 사업자 선택 시 체크리스트 (요약)

- **데이터 외부 전송 가능 여부**
  - 전면 가능 → Public Cloud 우선 검토
  - 전면 불가 → Private Cloud(OpenShift) 우선 검토
  - 일부 가능 → Public + Private 혼합 또는 Public Cloud 위 관리형 OpenShift(ROSA/ARO) 검토
- **기존 인프라 및 라이선스**
  - AWS 기반 시스템이 많은가?
  - Microsoft 365 / Azure AD / Windows 서버 비중이 높은가?
  - GCP/BigQuery 기반 데이터 파이프라인을 이미 운영 중인가?
  - OpenShift 클러스터 또는 Red Hat 서브스크립션을 이미 보유하고 있는가?
- **데이터 위치 및 규제**
  - 국내/특정 리전 저장 의무가 있는가?
  - 금융/공공 규제 등으로 특정 사업자/리전이 강제되는가?
- **조직의 운영 역량**
  - Kubernetes/EKS/GKE/OpenShift 운영 경험이 있는가?
  - GPU 노드 관리·모델 서빙 파이프라인을 자체 운영할 인력이 있는가?
- **우선순위**
  - 글로벌 리전 / 에지 네트워크 → Public Cloud
  - 엔터프라이즈 통합 (Microsoft 스택) → Azure
  - 데이터/분석 중심 아키텍처 → GCP
  - 데이터 격리·규제 충족 최우선 → Private Cloud(OpenShift)

위 체크리스트와 본 문서의 **사업자·플랫폼별 비교**를 기반으로,  
`2.2_배포환경유형과인프라제약.md`의 **클라우드 환경** 선택 이후,  
구체적인 Cloud 사업자 또는 Private Cloud 플랫폼을 체계적으로 결정할 수 있다.
